RENTGEN: Behavior-First API Testing Before Automation

A deterministic method for generating structured API evidence from a single known-good request

by Liudas Jankauskas

[First version: July 2026]
[rentgen.io]
pdf-version: [PDF]
www-version: [WWW]

Abstract

API testing commonly begins with a manually assembled request whose successful response is treated as evidence that an endpoint works. This establishes reachability, but it does not systematically examine input boundaries, type handling, authentication failure, protocol behavior, error semantics, response metadata, or basic performance signals. Traditional automation can cover these concerns, yet it normally requires test design, code, assertions, maintenance, and pipeline integration before useful evidence appears.

Rentgen is a local-first, behavior-first API testing system designed for the interval before a conventional automated suite exists. Starting from one known-good request, it derives structured mutations and protocol checks, executes them through a deterministic engine, and compares observed behavior with explicit, configurable expectations. The same project configuration can be retained, shared, and executed in the desktop application, command line, Docker, or continuous integration environments.

Rentgen does not prove functional correctness and is not a substitute for domain-specific tests, security assessment, or long-term regression automation. Its purpose is narrower: to expose observable API behavior quickly enough to improve decisions before teams invest in scripted automation.

This paper defines the problem Rentgen addresses, documents its operating model and test taxonomy, explains its interpretation rules and trust boundaries, reports field observations from public production APIs, and describes an experimental AI-directed extension under the Automation Before Automation (ABA) research program. The central design separation is deliberate: probabilistic systems may select what to explore, while execution and evidence evaluation remain deterministic and inspectable.

Introduction

API clients make it easy to send a request and inspect a response. They do not automatically establish how an API behaves when the assumptions encoded in that request are disturbed. A successful response to expected input gives a development team a useful starting point, but it supplies little evidence about malformed data, boundaries, optional fields, authorization failure, unsupported methods, oversized payloads, error handling, caching, security headers, or response-time behavior.

Conventional test automation can provide this evidence, but only after someone identifies scenarios, designs fixtures, writes assertions, controls environments, manages test data, and maintains the suite as the API changes. This creates a practical interval between the first successful request and the first maintainable automated suite.

Rentgen operates within this interval. It applies the Automation Before Automation (ABA) concept: automated exploration before automated verification. A known-good request becomes a seed from which Rentgen generates controlled variations, executes them, records the resulting behavior, and presents findings for review. The purpose is not to create an instant permanent regression suite. The purpose is to discover what the system actually does before deciding what deserves long-term automation.

The research question addressed by this paper is: How much structured and repeatable evidence about an API can be produced from one valid request before a team writes a conventional automated test suite?

1. The Testing Gap

A 2xx response to one expected input demonstrates only that a particular interaction was accepted at a particular moment. It does not establish that malformed inputs are rejected safely, that authorization failures are represented correctly, that unsupported methods are handled consistently, or that the endpoint remains stable across boundary and load conditions. HTTP distinguishes successful, client-error, and server-error classes; a 5xx response denotes a server-side failure condition rather than controlled rejection of an invalid request [1].

The practical gap is therefore not the absence of API clients. Teams already have effective ways to send requests. The gap is the distance between sending a request and obtaining broad, reviewable evidence without first building a bespoke test suite.

A durable automated suite is valuable precisely because it encodes product-specific expectations. That specificity also creates an activation cost: engineers must select scenarios, build fixtures, write assertions, control environments, manage data, and maintain the suite as the API evolves. When automation begins only after implementation is considered complete, easily discoverable protocol and validation defects may survive until a later and more expensive feedback stage.

Automation Before Automation names a distinct discovery phase: automated mechanisms generate observations and failure signals before stable regression tests are designed. ABA does not compete with scripted automation. It supplies earlier evidence that helps determine what deserves a permanent test, what is product-specific, and what is merely an implementation inconsistency [8].

Phase Primary question Typical artifact
Manual requestCan I reach the endpoint?A known-good cURL or equivalent request
Rentgen / ABAHow does it behave when assumptions are disturbed?Generated observations, findings, and project expectations

2. Design Principles

Rentgen is built around five constraints. First, a known-good request is treated as a seed, not a specification. Second, test derivation and execution are deterministic: the same project state and environment should result in the same intended checks. Third, expectations must be visible and configurable rather than hidden in opaque scoring. Fourth, generated findings are evidence for review, not automatic proof of a vulnerability or business defect. Fifth, sensitive request and response material remains local by design.

BEHAVIOR-FIRST: Rentgen asks what the API actually does under controlled variation. It does not infer that an endpoint is correct merely because a baseline request succeeds.

3. Operating Model

Input and normalization

The workflow begins with a request already known to succeed. In the desktop application this may be imported as cURL, then normalized into method, URL, headers, query parameters, body, authentication material, and environment variables. The baseline request is executed before derived cases so that later observations have an explicit point of comparison.


Field mapping

Request fields are mapped to test-relevant categories such as string, number, numeric string, email, phone, URL, IPv4, enum, bounded range, random value, or do-not-test. Mappings may be inferred and then adjusted by the user. This separates syntactic mutation from product knowledge: Rentgen provides reusable generators, while the user retains authority over field meaning and expected behavior.


Generation, execution, and evaluation

After mapping, the deterministic engine generates variations, executes them against the selected environment, evaluates observed behavior, and retains the evidence.

StageDeterministic responsibility
GenerateCreate field mutations and protocol-level variants from stored mappings and enabled checks.
ExecuteSend each derived request and record status, headers, body, size, and timing.
EvaluateCompare the observation with the configured expectation, normally an accepted 2xx/3xx outcome or a controlled 4xx rejection.
ReportExpose pass, fail, manual-review, ignored, timing, and request/response evidence without concealing disabled checks.
PersistStore mappings, expectations, ignored tests, collections, variables, and project state in the portable .rentgen project.

The simplified default model treats 2xx–3xx as accepted behavior, 4xx as controlled handling of a problematic request, and 5xx as a server failure signal. This is a heuristic, not a universal oracle. Rentgen v1.22 therefore allows expected 2xx or 4xx outcomes to be set per data-driven case and retained in the project.

4. Test Taxonomy

Data-driven input tests

Data-driven tests perturb one field or structural property while preserving the rest of the seed request. Categories include missing and null values, empty strings, leading and trailing whitespace, casing changes, invalid types, malformed payloads, boundary values, enum variation, oversized strings, and format-specific invalid values. The objective is not random traffic. Each mutation corresponds to a recognizable assumption about validation or parsing.

CategoryExample mutationSignal sought
PresenceRemove a field or send nullRequired/optional behavior and controlled rejection
RepresentationReplace number with string or objectType enforcement and parser resilience
BoundaryUse min-1, min, max, max+1Off-by-one behavior and range enforcement
NormalizationAdd whitespace or alter caseCanonicalization and silent data-corruption risk
Domain formatInvalid email, IPv4, URL, or numeric stringField-specific validation behavior
SizeOversized field or bodyPayload controls and graceful failure
StructureMalformed JSON or unexpected collection shapeParser and error-handler stability

Protocol and security-hygiene checks

Rentgen includes observable checks for missing and invalid authorization material, OPTIONS behavior and Allow discovery, unsupported methods, non-existent routes, CORS behavior, reflected payload safety, uppercase domain and path handling, large payload behavior, server-version exposure, clickjacking controls, HSTS, MIME-sniffing protection, and cache control for private responses. These checks overlap with concerns identified by the OWASP API Security project, including resource consumption and security misconfiguration [2] [3]. They are screening checks rather than a penetration test.

Check familyIllustrative expectationInterpretation boundary
AuthenticationMissing or invalid credentials produce 401A different status may be intentional; business authorization is not proven.
MethodsUnsupported method produces 405 or 501Gateway policy may alter the externally observed response.
DiscoveryOPTIONS returns 200/204 with AllowOPTIONS support varies; absence is a signal, not automatically a defect.
Payload sizeOversized request produces 413Infrastructure may enforce the limit before the application.
HeadersHSTS, nosniff, frame protection, private cache rulesApplicability depends on deployment and response type.
RoutingUnknown route produces 404; casing is consistentFramework and security policy may intentionally conceal resources.

Performance insights

Rentgen produces lightweight performance signals rather than a capacity certification. Current checks include repeated-request timing, median and percentile-oriented observations, response size, ping latency, estimated network share, load-test observations, and detection of collection responses without visible pagination parameters. These signals help distinguish obvious latency, payload, or pagination risks that merit dedicated performance work.

NOT A LOAD-TESTING CLAIM: A quick generated run can reveal suspicious behavior; it cannot establish production capacity, saturation limits, or service-level compliance without a controlled performance model.

5. Portable Project Configuration

A .rentgen project is the bridge between discovery and repeatability. It can contain requests, collections, static and dynamic environment variables, history, field mappings, expected results, and ignored checks. Because configuration travels with the project, a reviewer can see both what was tested and which expectations were intentionally changed or disabled. This makes the artifact suitable for source control or another team-controlled storage mechanism without requiring hosted synchronization.

The desktop application supports interactive configuration and review. The CLI executes exported projects locally, through Docker, or in CI/CD, with collection and environment selection and machine-readable JSON reporting. The same conceptual test definition therefore moves from exploratory use to repeatable pipeline execution without rewriting it in another framework.

6. Local-First Trust Model

API requests frequently contain credentials, personal data, internal hostnames, proprietary schemas, and regulated information. Rentgen treats locality as an architectural constraint: the desktop tool does not require accounts, hosted synchronization, telemetry, or a cloud execution service. Requests, responses, projects, and generated observations remain under the operator's control. This allows use in environments where SaaS API tooling or external AI is restricted.

Local-first does not mean risk-free. Operators remain responsible for endpoint authorization, test-data handling, credential storage, side effects, traffic volume, and the security of the host machine and project files. Generated tests should be run only against systems the operator is authorized to test. Destructive or financially consequential endpoints require explicit controls beyond generic test generation.

7. Evidence and Field Observations

Rentgen has been exercised against public production APIs including ChatGPT, Amazon, Jira, Proton Pass, Revolut, Trello, Azure DevOps, and Stripe. Published case narratives report observations such as malformed-input 500 responses, late payload rejection, HTML returned from a JSON API, unexpected method handling, authorization inconsistencies, missing response protections, and high latency. The ChatGPT case led to fixes after responsible reporting [7].

These cases demonstrate feasibility and defect-discovery potential, but they do not constitute a controlled comparative evaluation. Different endpoints, configurations, time periods, and authorization contexts were used. The findings should therefore be treated as field observations, not as evidence that one vendor or API is generally less secure or lower quality than another. Reproducible benchmark research remains future work.

Evidence classWhat it supportsWhat it does not support

8. Relationship to Existing Testing

Rentgen replaces neither API clients nor programmable test frameworks. Postman, Playwright, REST Assured, contract-testing tools, security scanners, and load-testing systems address broader or deeper workflows. Rentgen occupies the earlier moment when a working request exists but a maintained test model does not. Its output can reveal where product-specific assertions are needed and which generated checks are stable enough to become continuous gates.

The practical sequence is cumulative: use Rentgen to discover and triage observable behavior; fix or accept the findings; encode important business rules in a durable suite; and use CI/CD to enforce the behaviors whose long-term value exceeds their maintenance cost. NIST's Secure Software Development Framework likewise frames verification as part of a broader secure development discipline, not as a single-tool outcome [4].

Build or change an endpoint → Run Rentgen → Review and configure findings → Create maintainable automation → Enforce through CI/CD

9. Limitations and Validity Threats

Rentgen produces structured evidence, but its coverage and conclusions remain bounded by the supplied request, configured environment, observable interface, and generic expectations.

10. AI-Directed Discovery: Boruzele

Boruzele is an experimental exploratory-testing agent within the Rentgen.io research initiative. It observes an application UI, builds a compact model of interactive elements, asks a language model to select the next interaction, executes that interaction deterministically through Playwright, and records console errors, HTTP 5xx responses, navigation state, and action history. The loop repeats until a configured stopping condition [9].

The intended Rentgen platform architecture extends this separation across UI and API layers. A language model may decide which path, action, or API behavior deserves exploration. Rentgen then generates and executes deterministic tests and evaluates the resulting evidence. Observations return to the decision layer, which chooses the next exploration target. The model therefore guides discovery but does not become the final oracle.

LayerResponsibilityTrust characteristic
ObservationExtract UI/API context and prior signalsDeterministic collection
DecisionSelect the next exploratory objective or interactionProbabilistic and replaceable LLM
ExecutionPerform UI action and/or generated API testsDeterministic Playwright/Rentgen engine
EvaluationApply explicit expectations and retain evidenceInspectable deterministic rules
Learning loopFeed discoveries into subsequent decisionsBounded by goals, steps, and policy
SAFETY SEPARATION: Use AI to choose where to look. Use deterministic mechanisms to execute, measure, and report what happened.

This architecture is research in progress. It raises open questions about exploration coverage, hallucinated intent, action safety, reproducibility, model cost, stopping criteria, and the transition from transient discoveries to maintainable tests. Those questions should be evaluated empirically rather than hidden behind an AI-powered testing label.

11. Research Agenda

The next research stage should move from case narratives toward reproducible measurement. A useful benchmark would provide versioned APIs with seeded defects across validation, authentication, protocol semantics, payload handling, response headers, latency, and state transitions. Rentgen could then be compared with a baseline manual request, generic fuzzing, and scripted tests using time-to-first-signal, defect-class recall, false-positive review cost, execution reproducibility, and maintenance effort.

For AI-directed ABA, experiments should compare models and policies on path diversity, unique signal discovery, repeated-action rate, invalid-action rate, cost, and deterministic replay. Importantly, the evaluation should separate decision quality from execution quality: an LLM may choose an unproductive path while the deterministic engine still performs the selected operation correctly.

12. Conclusion

Rentgen formalizes a practical but underserved stage of API development: the period after a request first works and before a maintained automated suite exists. By expanding one known-good request into structured, configurable, and portable checks, it produces early evidence about validation, protocol behavior, security hygiene, and performance signals without requiring scripts or cloud execution.

Its contribution is not a universal test oracle. It is a disciplined separation of concerns: the seed request supplies access, mappings supply test meaning, deterministic generators supply variation, explicit expectations supply evaluation, and human review supplies context. The same project can then move into CLI, Docker, and CI/CD execution.

Under the wider ABA research program, probabilistic AI can direct exploration while Rentgen and Playwright retain deterministic responsibility for execution and evidence. This preserves the creative reach of model-guided discovery without surrendering test results to an opaque model judgment.
